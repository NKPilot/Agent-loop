# 需求：loopAI

**定义日期:** 2026-05-27
**核心价值:** 让 AI Agent 不仅"能跑"，而且可靠、可观测、可扩展——从 harness 设计的深度思考出发，构建值得信任的 agent 系统。

## v1 需求

首次发布的需求。每条需求映射到对应 Roadmap 阶段。

### Agent 核心循环

- [ ] **CORE-01**: 实现 ReAct 状态机（REASON → ACT → OBSERVE → FINISH → ERROR），而非简单的 while 循环
- [ ] **CORE-02**: 通过 OpenAI 兼容 API 调用 LLM（可配置 base_url、api_key、model）
- [ ] **CORE-03**: 流式输出 agent 每步的思考、调用和观察结果（async generator/SSE）
- [ ] **CORE-04**: 步骤预算 + 终止条件（目标达成 / 不可达成 / 预算耗尽 80% 预警）
- [ ] **CORE-05**: 消息结构交替校验（tool_call 和 tool_result 必须成对，防止孤立的 tool call 导致幻觉）
- [ ] **CORE-06**: 基础循环检测（同一工具连续调用 3 次以上触发干预）
- [ ] **CORE-07**: 从第一轮即开启 JSONL 日志记录，格式化为结构化事件

### 工具系统

- [ ] **TOOL-01**: 通过 `@tool` 装饰器将 Python 函数注册为 Agent 工具，自动从 type hints + docstring 生成 JSON Schema
- [ ] **TOOL-02**: 工具执行管道：参数校验 → 沙箱执行 → 结果结构化 → 注入上下文
- [ ] **TOOL-03**: Bash/Shell 工具——安全的 subprocess 执行（`shell=False`，数组参数，超时控制）
- [ ] **TOOL-04**: 危险命令确认机制——rm、dd、mkfs 等操作在执行前暂停并要求确认
- [ ] **TOOL-05**: 命令权限分级（safe / moderate / dangerous），每级不同的执行策略
- [ ] **TOOL-06**: 错误分类体系——至少 4 类（Transient / ToolExecution / GuardViolation / Fatal），不同类别不同恢复策略
- [ ] **TOOL-07**: 瞬时错误的基础重试——指数退避 + 随机抖动

### 上下文管理

- [ ] **CTX-01**: Token 计数——通过 tiktoken 实时追踪上下文用量
- [ ] **CTX-02**: 上下文压缩——达到 75% 窗口阈值时触发压缩（工具输出截断、旧消息摘要、过期历史移除）
- [ ] **CTX-03**: 追加式消息存储——不做原地修改，压缩产生的新消息追加到末尾
- [ ] **CTX-04**: 溢出文件存储——超长工具输出（>80K 字符）写入溢出文件而非截断

### 韧性与恢复

- [ ] **RES-01**: 检查点管理器——关键节点状态序列化 + 崩溃恢复
- [ ] **RES-02**: 循环检测升级——基于分类的干预策略，附元认知提示
- [ ] **RES-03**: 失败注册表——持久化"不再重复"的操作列表
- [ ] **RES-04**: 守卫阶段管道——token 预算守卫、成本守卫、速率限制守卫
- [ ] **RES-05**: 分层自愈恢复——外观修复 → 上下文内重试 → 完整重试 + 退避 → 人工升级
- [ ] **RES-06**: 熔断器——某工具连续失败后暂停该工具

### 可观测性与 Web 前端

- [ ] **OBS-01**: 事件总线——所有 harness 层通过结构化事件向外报告状态
- [ ] **OBS-02**: SSE 实时流端点——将 agent 状态变更实时推送到前端
- [ ] **OBS-03**: React Web 前端——Agent 时间线、思考步骤、工具调用卡片（含状态标记）
- [ ] **OBS-04**: Token/成本追踪——在前端展示每次调用的 token 消耗
- [ ] **OBS-05**: 会话历史浏览——查看过往 agent 会话记录

### 业务验证场景

- [ ] **BIZ-01**: 磁盘空间诊断与清理——agent 能扫描磁盘、定位大文件、分析安全性、请求用户确认后清理
- [ ] **BIZ-02**: 可交互演示——通过网页、CLI 或 Notebook 完整演示上述流程

## v2 需求

延迟到后续版本。

### 高级特性

- **ADV-01**: 会话回放——在前端按步前进/后退回放历史 agent 会话
- **ADV-02**: 人在回路通用暂停/恢复——危险操作之外也支持任意节点暂停
- **ADV-03**: 类型化状态 + Reducer——取代 ad-hoc 消息列表
- **ADV-04**: 工具推测执行——预执行可能的下一步工具调用（实验性）
- **ADV-05**: 因果溯源 / Ablation 分析——衡量每个工具调用对最终结果的贡献
- **ADV-06**: 工具 Fuzz 测试——自动发现工具调用的边界用例

### 扩展

- **EXT-01**: 多 LLM 提供商适配（Anthropic、Gemini 等）
- **EXT-02**: 多 Agent 协作（Agent 作为工具调用模式）
- **EXT-03**: 长期记忆 / 向量数据库

## 不在范围内

| 功能 | 原因 |
|------|------|
| 完整多 Agent 编排 | 稀释 harness 深度关注点；v2 考虑 Agent-as-tool 模式 |
| 50+ LLM 提供商的兼容代理层 | 维护负担过大；先做好 OpenAI 兼容接口 |
| Agent 自主安装工具包 | 安全风险——任意包安装是逃逸向量 |
| 可视化 Agent 构建器（拖拽式） | 目标用户是开发者，CLI + YAML 配置即可 |
| 自动任务分解 | 不可预测的子任务结构，难以观察和审计 |
| 长期记忆 / 向量数据库 | 跨度太大，先做好会话级上下文管理 |
