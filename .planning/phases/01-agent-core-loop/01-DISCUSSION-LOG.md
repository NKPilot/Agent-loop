## 状态机设计

| 选项 | 描述 | 选中 |
|------|------|------|
| REASON → FINISH | LLM 判断无需工具时直接从 REASON 跳到 FINISH——标准 ReAct 行为 | |
| 仍需一轮 OBSERVE | 始终走完整路径，但无工具时 ACT 为空中转 | |

**用户选择:** Other — 询问"有什么成熟的做法吗"
**结论:** 采用 REASON → FINISH 直跳。依据: OpenAI function calling 原生行为（有 tool_calls 则执行，无则结束）、ReAct 原始论文中 Finish[answer] 即为一种 Action、Anthropic tool use 的 stop_reason 机制。

## 流式输出粒度

### 事件粒度

| 选项 | 描述 | 选中 |
|------|------|------|
| 步骤级事件 | 每进入新状态发一个事件，简单但不含 token 级实时输出 | |
| 步骤 + Token级 | 步骤事件之外 LLM 输出文本也逐 token 推送 | ✓ |

### 事件组织方式

| 选项 | 描述 | 选中 |
|------|------|------|
| 扁平事件流 | 所有事件平铺，消费端按 type 字段过滤 | |
| 分层事件流 | 顶层生命周期包裹内层子事件流 | ✓ |

### 发射机制

| 选项 | 描述 | 选中 |
|------|------|------|
| Async Generator | Python 原生，单一消费者 | |
| Event Bus / Pub-Sub | asyncio.Queue 驱动，支持多消费者同时订阅 | ✓ |

**用户选择:** 发射机制选择"你推荐什么呢"
**结论:** Event Bus。理由: 当前就需要 3 个消费者（CLI、JSONL Logger、SSE），async generator 只支持单消费者，后期重构成本高。实现用 asyncio.Queue 驱动的轻量 pub-sub，几十行代码，零外部依赖。

## 步骤控制与终止

### 默认步骤预算

| 选项 | 描述 | 选中 |
|------|------|------|
| 15-20步 | 满足磁盘诊断场景，大多数交互式 agent 的默认范围 | ✓ |
| 30-50步 | 更宽松但成本更高 | |

### 预算耗尽行为

| 选项 | 描述 | 选中 |
|------|------|------|
| 强制终止+摘要 | 最后一轮总结机会后终止 | ✓ |
| 立即终止 | 达到上限立即结束 | |

### 不可达成判定

| 选项 | 描述 | 选中 |
|------|------|------|
| LLM 自判 | 依靠 LLM 语义理解 | |
| 系统规则检测 | 确定性信号检测 | |
| 两者结合 | 硬信号 + LLM 声明，覆盖面最全 | ✓ |

### 80% 预警

| 选项 | 描述 | 选中 |
|------|------|------|
| 注入提醒提示 | 在 80% 时向 LLM 上下文注入提示 | ✓ |
| 仅日志记录 | 不干预 agent 行为，交给外层处理 | |
