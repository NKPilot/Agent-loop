<interfaces>
BudgetGuard (from RESEARCH.md Pattern 5 + D-06, D-07, D-09):
- Constructor: max_steps: int = 15, warn_pct: float = 0.80
- Method: check(step_count: int, messages: list[dict]) -> tuple[bool, list[dict], str | None]
- Returns: (should_continue, modified_messages, action) where action is None/"warn"/"final"
- At warn threshold (80%): inject system message with percentage + remaining steps
- At exhaustion: inject final summary prompt, return action="final"
- D-08 unreachable detection: system detects consecutive failures >= 3 -> signal unreachable

LoopDetector (from RESEARCH.md Pattern 4 + CORE-06):
- Constructor: window_size: int = 20, warn_threshold: int = 3, block_threshold: int = 5
- Static method: _signature(tool_name: str, arguments: dict) -> str using hashlib.sha256
- Method: check(tool_name: str, arguments: dict) -> tuple[bool, str] returns (should_proceed, action)
- action values: "allow", "warn", "block" (at 5 consecutive), "force_exit" (pattern persists after block)

MessageValidator (from RESEARCH.md + CORE-05):
- Strict validation per RESEARCH open question #2 recommendation
- Check: every assistant message with tool_calls has a corresponding tool-role message for each tool_call_id
- On violation: raise ValidationError with specific message about which tool_call_id is orphaned

| Boundary | Description |
|----------|-------------|
| Environment -> Config | Untrusted env var values become AgentConfig fields. pydantic validation constrains types. |
| LLM output -> MessageValidator | Messages from LLM (tool_calls structure) validated before next API call. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | AgentConfig.api_key | mitigate | SecretStr in pydantic model; custom __repr__ hides value; never logged or serialized |
| T-02-02 | Denial of Service | BudgetGuard | mitigate | step_count >= max_steps forces "final" action regardless of LLM behavior |
| T-02-03 | Spoofing | MessageValidator | mitigate | Strict orphan detection prevents tool_result injection without a corresponding tool_call |
| T-02-04 | Elevation of Privilege | LoopDetector._signature | accept | sha256 used only for deterministic comparison, not security; args from LLM tool calls trusted |
</threat_model>
