    Step 2: 创建 `src/loopai/context/__init__.py`，导出 TokenCounter 和 TokenizerProtocol。

    Step 3: 创建 `src/loopai/context/token_counter.py`，包含：

    3a. 定义 `TokenizerProtocol` 作为 typing.Protocol，声明方法签名：
        - `def count_text(self, text: str) -> int`
        - `def count_message(self, message: dict) -> int`
        - `def count_messages(self, messages: list[dict]) -> int`
      这是 D-04 预留的 provider-tokenizer 接口。

    3b. 定义 `TokenCounter` 类，实现 TokenizerProtocol：
        - `__init__(self, encoding_name: str = "cl100k_base")`：初始化 tiktoken 编码器
        - `count_text(self, text: str) -> int`：对文本做 tiktoken 编码并返回 token 数
        - `count_message(self, message: dict) -> int`：按 OpenAI 消息格式计算单条消息 token 数。格式参考 OpenAI 官方计数规范：
          * 基础开销：3 tokens（role 标记）
          * content 文本：count_text(content)
          * name 字段：1 token 额外开销 + count_text(name)
          * tool_calls：3 tokens 基础 + 每条 call：工具名称和 arguments 的 token 数
        - `count_messages(self, messages: list[dict]) -> int`：对所有消息求和，末尾加 3 tokens（assistant 角色粘合开销）

    Step 4: 创建 `tests/test_token_counter.py`，包含测试类：
        - test_count_text_ascii：验证纯英文文本计数
        - test_count_text_chinese：验证中英文混合计数
        - test_count_message_with_content：验证单条用户消息计数
        - test_count_message_with_tool_calls：验证含 tool_calls 的 assistant 消息计数
        - test_count_messages_full_list：验证完整消息列表计数（至少包含 user + assistant + tool 三条）
        - test_token_counter_protocol：验证 TokenCounter 实现了 TokenizerProtocol
    
    不要使用 `uv add tiktoken` 命令 — 直接在 pyproject.toml 中编辑即可。

  <action>
    修改 `src/loopai/tools/executor.py`：

    1. 移除常量 `_MAX_RESULT_BYTES = 100 * 1024`，新增两个常量：
       - `_OVERFLOW_THRESHOLD_CHARS = 80 * 1024`  # 80K 字符溢出阈值
       - `_OVERFLOW_DIR = ".sandbox/overflow"`    # 溢出文件根目录

    2. 修改 `execute()` 方法签名，新增可选参数：
       - `session_id: str = ""` — 用于溢出文件命名
       - `tool_call_id: str = ""` — 用于溢出文件命名
       将这些参数通过 `_execute_with_retry` 和 `_execute_once` 传递。

    3. 在 `_execute_once` 中新增 `_write_overflow_file(result_str)` 私有方法：
       - 用 `os.makedirs(_OVERFLOW_DIR, exist_ok=True)` 确保目录存在
       - 文件路径格式：`.sandbox/overflow/{session_id}_{tool_call_id}_{timestamp}.txt`
         （只有当 session_id 非空时使用完整格式，否则回退到 `{timestamp}.txt`）
       - 用 UTC 时间戳 `datetime.now(timezone.utc).strftime("%Y%m%dT%H%M%S")` 生成文件名部分
       - 用 `pathlib.Path.write_text()` 写入结果字符串（UTF-8）
       - 返回文件路径字符串

    4. 修改 `_execute_once` 中的截断逻辑：删除原有的 100KB 截断代码块
       ```python
       # DELETE this block entirely:
       if isinstance(data, str) and len(data.encode("utf-8")) > _MAX_RESULT_BYTES:
           ...
           overflow_file = None  # Could write to temp file in future
       ```
       替换为溢出文件逻辑：
       ```python
       if isinstance(data, str) and len(data) > _OVERFLOW_THRESHOLD_CHARS:
           overflow_file = self._write_overflow_file(tool_name, session_id, tool_call_id, data)
           # data 保持完整 — FSM._handle_act 负责在注入上下文时替换为引用
       ```

    5. 新增导入：`os`, `datetime`, `pathlib.Path`（将 `from pathlib import Path` 加到顶部）
       注意：`time` 已导入（用于 duration 计时），`inspect` 已导入。

    - 在现有 `TestToolExecutor` 类中或新增测试类：
      - `test_overflow_file_created`：execute() 一个产生 >80K 字符输出的工具，验证 ToolResult.overflow_file 非空，且文件实际存在于磁盘
      - `test_no_overflow_below_threshold`：execute() 一个产生 <80K 输出的工具，验证 overflow_file 为 None
      - `test_overflow_file_content`：验证溢出文件中内容与工具输出完全一致

| ToolExecutor -> overflow file | 大型字符串写入磁盘文件，文件路径基于 session_id + tool_call_id |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01-01 | E (Elevation of Privilege) | overflow file write | mitigate | 溢出文件路径仅使用 session_id（UUID）和 tool_call_id 拼接，不允许路径穿越字符。工具输出内容直接写入文件，不做 eval/exec |
| T-03-01-02 | I (Information Disclosure) | overflow file content | accept | 溢出文件存储在本地 .sandbox/overflow/，权限 0o644。生产环境中需确保 .sandbox 不在 web 可访问路径下 |
